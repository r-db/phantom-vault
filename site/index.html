The new threat model

Old vaults guard the door. You invited the threat inside.

Every secret manager was built to stop outsiders. But you handed an AI agent the power to read your files, run your commands, and watch the output. The threat isn't at the door anymore.

Without Phantom Vault

The AI reads everything

Grant an assistant shell access and it can see your .env, echo your keys, and pipe them anywhere. One leaked key is a drained account at 3am.

  • Keys sit in plaintext env vars
  • Prompt injection can exfiltrate them
  • Secrets land in logs, output, history
With Phantom Vault

The AI works through the vault

Secrets are injected into the subprocess, and the value comes back as [REDACTED] — the AI uses the secret without the raw string in its output.

  • Encrypted at rest, master-password locked
  • Called by name via vault_run
  • Every use written to the audit log
How it's designed to work

What happens when a secret is needed.

under audit
01 · analyze

Pre-flight the command

Command pre-analysis (in the codebase; being hardened) checks for oracle and side-channel tricks before anything runs.

verified
02 · inject

Slip the key in

The secret is placed into the subprocess environment for that one command — not printed, not echoed.

under audit
03 · sanitize

Scrub the output

Output is filtered for the value and its encoded variants (base64, hex, URL) before returning. Effectiveness is under active audit.

under audit
04 · audit

Record the access

Every access is written to the audit log. Tamper-evident chaining and canary detection are implemented and being verified.

Verified means proven in the code today. Under audit means it exists but an outside checker (not us) is still confirming it holds — until then we don't claim it as fact.

What's proven, and what's not

The honest security status.

It's open source under Apache 2.0 — read every line. Each item below is labeled verified (proven in the code) or under audit / planned (implemented but not yet independently confirmed). We understate on purpose.

verified

AES-256-GCM at rest

Secrets encrypted at rest; master password derived with Argon2; keys zeroized in memory.

verified

MCP server & CLI

A 14-command CLI and an MCP server. Claude Code calls vault_run; the value comes back as [REDACTED].

verified

Access audit log

Every access to a secret is recorded. Tamper-evident HMAC chaining is under audit.

under audit

Output sanitization

Filters the value and its encoded variants from returned output. Implemented; effectiveness being independently verified.

under audit

Canary secrets

Decoy credentials meant to flag exfiltration attempts. Implemented; not yet proven effective.

planned

Hardware backing

Planned: Secure Enclave, TPM 2.0, and FIDO2 as an extra layer where available.

Install it in one line.

No account, no credit card, no telemetry. Riscent built this and gave it away.

$ curl -fsSL https://phantomvault.riscent.com/install | bash
Prefer Cargo? cargo install phantom-vault Verify: phantom-vault --version
Then, four commands
  1. 1
    Create your vault (encrypted, on your machine)
    phantom-vault init
  2. 2
    Add a secret — you're prompted for the value, it's never echoed
    phantom-vault set STRIPE_KEY
  3. 3
    Run any command — the secret is injected, the output scrubbed
    phantom-vault run -- ./deploy.sh
  4. 4
    Connect it to your AI assistant (Claude Code, Cursor, …)
    phantom-vault mcp serve
Or let your AI do it

Paste this into Claude Code, Cursor, or any AI CLI.

It installs Phantom Vault, wires up the MCP server, and shows you how to use it — the AI-native way to onboard a tool built for AI.

Set up Phantom Vault for me — the free, open-source vault that lets you
use my API keys and secrets without ever seeing them.

1. Install it: run
     curl -fsSL https://phantomvault.riscent.com/install | bash
   (or `cargo install phantom-vault` if Rust is available), then confirm
   with `phantom-vault --version`.
2. Initialize my vault: `phantom-vault init`.
3. Register Phantom Vault's MCP server with this assistant so you can run
   commands that use my secrets through `vault_run`. The server starts with
   `phantom-vault mcp serve` — add it to my MCP config for this tool.
4. Show me how to add a secret with `phantom-vault set <NAME>`, then prove
   it works: store a test secret, run a command that uses it, and confirm
   the value never appears in your output.
5. Give me a 3-line cheat sheet for day-to-day use.

Never print my secret values. Docs: https://github.com/r-db/phantom-vault
